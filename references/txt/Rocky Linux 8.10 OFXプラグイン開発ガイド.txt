Rocky Linux 8.10におけるOpenFXプラグイ
ン開発：エンジニアリング・詳細ガイド
ビジュアル・エフェクト（VFX）およびポストプロダクション業界において、ソフトウェアの相互運用性は
長年の課題であった。この課題を解決するために提唱された標準規格がOpenFX（別名OFX）であ
る。OpenFXは、2Dビジュアル・エフェクトやコンポジッティング・プラグインのためのオープン標準で
あり、この規格に準拠して書かれたプラグインは、規格をサポートするあらゆるホストアプリケーショ
ン上で動作することが可能となる 1。本報告書では、エンタープライズLinuxの堅牢な基盤を持つ
Rocky Linux 8.10を対象とした、OpenFXプラグイン開発の詳細な技術ガイドを提供する。

Rocky Linux 8.10は、Red Hat Enterprise Linux (RHEL) 8系列の最終マイナーリリースであり、今後5
年間にわたるセキュリティアップデートが保証されているため、プロフェッショナルな制作環境におけ
る開発プラットフォームとして極めて高い信頼性を有している 2。本ガイドでは、環境構築からAPIアー
キテクチャ、レンダリング処理、マルチスレッド最適化、そしてパッケージングに至るまで、専門的な
知見に基づいた詳細な解説を行う。


開発環境の構築とシステム基盤
Rocky Linux 8.10での開発において、まず考慮すべきはコンパイラ・ツールチェーンとライブラリの整
合性である。VFX業界では「VFX Reference Platform」と呼ばれる標準規格が毎年更新されており、
開発環境はこのガイドラインに準拠することが望ましい 4。

コンパイラおよびビルドツールの選定
Rocky Linux 8.10のデフォルトのGCCバージョンは8.5.0であるが、現代的なOpenFX開発において
はC++17以降の言語機能が要求されることが多い 6。これに対応するため、AppStreamリポジトリ経
由で提供される「GCC Toolset」を利用することが標準的である。GCC Toolsetは、システムの基本ラ
イブラリを破壊することなく、最新のGCC、GDB、およびパフォーマンス解析ツールを提供するため
の仕組みである 8。

OpenFXアーキテクチャの核心
OpenFXの設計思想は、ホスト（Host）とプラグイン（Plug-in）の間の「疎結合」にある。ホストはプラグ
インに対して機能のセットである「スイート（Suites）」を公開し、プラグインはホストからの「アクション（
Actions）」に応答することで動作する 1。

スイート（Suites）とアクション（Actions）のメカニズム
APIの実体はC言語の構造体であり、ホストは関数ポインタのテーブルをプラグインに渡す。これによ
り、C++のシンボルマングリング問題やABIの不整合を回避し、異なるコンパイラでビルドされたアプ
リケーション間での動的リンクを可能にしている 19。

アクションはC文字列（const char *）として定義されており、プラグインのメインエントリポイント関数
に渡される。この設計により、将来的なAPIの拡張において新しいアクションが追加されたとしても、
古いプラグインとの後方互換性を維持することが容易となっている 19。

エントリポイントの定義
OpenFXプラグインは、共有ライブラリ（DSO）としてビルドされ、ホストに対して以下の2つのシンボル
を公開しなければならない。
 1.​ OfxGetNumberOfPlugins: ライブラリに含まれるプラグインの数を返す。
 2.​ OfxGetPlugin: インデックスに対応するプラグインの構造体（OfxPlugin）を返す。

プラグイン構造体には、APIの名称、バージョン、およびメインエントリポイント関数のポインタが含ま
れる 19。メインエントリポイントは以下のシグネチャを持つ。



  C

この関数内で、渡されたaction文字列に応じた条件分岐を行い、必要な処理を実行することがプラ
グイン開発の基本構造となる 19。


プラグインのライフサイクルと初期化処理
OpenFXプラグインは厳格なライフサイクルに従って動作する。このプロセスの各段階を正しく実装す
ることは、メモリリークの防止や安定性の確保において極めて重要である。

ロードと記述フェーズ
バイナリがメモリにロードされると、最初にkOfxActionLoadが呼び出される。ここでは、グローバルな
データ構造の初期化や、ホストから提供される基本スイート（Property Suite, Parameter Suiteなど）
の取得が行われる 21。

続いて実行されるkOfxActionDescribeは、プラグインがその「正体」をホストに宣言する段階であ
る。このアクションでは、プラグインがサポートするピクセル深度（8ビット整数、16ビット整数、32ビット
浮動小数点など）、コンポーネント（RGBA, Alphaのみ等）、およびマルチスレッド性能について記述
する 19。ここでの記述は「記述子（Descriptor）」に対して行われ、実際の個別のエフェクト設定とは独
立している 19。

コンテキストに応じた記述
OpenFXには「コンテキスト（Context）」という概念があり、プラグインがどのような環境で使用される
かを定義する。例えば、「Filter」コンテキストは1つの入力と1つの出力を持ち、「General」コンテキス
トは任意の数の入力をサポートする 26。

画像処理の基礎とメモリ管理
プラグインの主目的は画像データの変換である。OpenFXにおける画像処理は、ピクセルデータへ
のダイレクトアクセスと、座標系の適切な理解が不可欠である。

クリップ、イメージ、ピクセルデータ
「クリップ（Clip）」は画像のストリームを表し、実際の画像データは「イメージ（Image）」として取得され
る。レンダリング中、プラグインはclipGetImage関数を介して特定の時間のイメージをフェッチする 23
。取得されたイメージはプロパティセットであり、ピクセルデータへのポインタ（kOfxImagePropData）
や、各走査線のバイト幅（kOfxImagePropRowBytes）などの属性を保持している 22。

合、メモリ上の配置は以下のようになる。

座標系と空間的認識
OpenFXは、解像度に依存しない「カノニカル座標系（Canonical Coordinate System）」を採用してい
る 23。これにより、異なる解像度の素材を組み合わせる際の計算が簡略化される。また、「ピクセル・
アスペクト比（Pixel Aspect Ratio）」を考慮することで、アナモルフィック・レンズで撮影された素材など
の歪みを正しく処理できる 22。

マルチスレッドレンダリングと最適化
現代のVFXプロダクションにおいて、CPUの全コアを活用した並列処理は必須要件である。OpenFX
は、ホスト主導とプラグイン主導の両面でマルチスレッドをサポートしている。

スレッドセーフティの宣言
プラグインは記述段階で、自身のスレッド安全性のレベルをホストに通知する。これにより、ホストは
どのようにレンダリングアクションをスケジュールするかを決定する。
     1.​ kOfxImageEffectRenderUnsafe: 同時に一つのインスタンスしかレンダリングできない。古い
         ライブラリを使用する場合などに指定する 23。
     2.​ kOfxImageEffectRenderInstanceSafe: 異なるインスタンスであれば並列実行可能。パラ
         メータ変更とレンダリングが別スレッドで行われる可能性がある 23。
     3.​ kOfxImageEffectRenderFullySafe: 同一インスタンス内でも、異なるRender Windowであれ
         ば完全に並列実行可能 23。

OfxMultiThreadSuiteの活用
プラグイン内部で並列化を行う場合、ホストが提供するOfxMultiThreadSuiteを利用することが推奨
される 29。このスイートは、OSレベルのスレッド生成を抽象化し、ホストが管理するスレッドプールを
利用してSMP（対称型マルチプロセッシング）実行を可能にする。

典型的な実装パターンでは、画像のY方向の走査線をスレッド数に応じて分割し、各スレッドに割り
当てる。



     C++



void multiThreadFunction(unsigned int threadId, unsigned int nThreads, void *customArg) {​
     // 全体のY範囲をnThreadsで分割し、threadIdに対応する範囲を処理する [29, 31]​
}​

この手法により、Rocky Linux 8.10が動作するマルチコア・ワークステーションの性能を最大限に引き
出すことができる。また、クリティカルセクションを保護するために、スイート内のmutexCreate、
mutexLock等を利用した排他制御も可能である 29。


パラメータ・システムとプロパティの管理
プラグインの柔軟性は、ユーザーが調整可能な「パラメータ」によって決まる。OpenFXのパラメータ
システムは、強力なプロパティ・メカニズムに基づいている。

パラメータの定義とUI構成
パラメータは整数、浮動小数点、文字列、ブール値、プッシュボタン、カスタムタイプなど多岐にわた
る。これらはkOfxActionDescribeInContext内で定義され、階層構造（Group）やページ（Page）に
よって整理される 32。



 パラメータ属性              役割                   設定例


 Label                UIに表示される名称           "Contrast", "Saturation" 32

 Hint                 ユーザー向けの説明文           "Adjusts the brightness
                                           difference" 19

 Default              初期値                  1.0, 0, false 19

 Range                有効な値の範囲              0.0 to 10.0 19

インタラクティブな応答
ユーザーがUI上でパラメータを変更すると、ホストはkOfxActionInstanceChangedアクションを送信
する。プラグインはこの通知を受け取り、他のパラメータの表示/非表示を切り替えたり、値を動的に
再計算したりすることができる 28。例えば、あるチェックボックスがオンになった時だけ詳細設定を表
示するといった、動的なユーザーインターフェースの構築が可能である 22。


高度な機能：GPUレンダリングとインターアクト
OpenFXはCPUレンダリングにとどまらず、ハードウェア加速や高度なユーザー・インタラクションをサ
ポートするための拡張を備えている。

CUDAおよびOpenCLによるGPU加速
高負荷な画像処理において、GPUの活用は劇的な速度向上をもたらす。OpenFX 1.5以降、CUDA、
OpenCL、MetalなどのGPU APIとの親和性が高められている 1。
●​ CUDA: ホストはプラグインに対し、cudaStream_tポインタを渡す。プラグインは自身のカーネル
   をこのストリーム上で実行し、非同期に処理を完結させることが期待される 24。
 ●​ OpenCL: 同様に、OpenCLコマンドキューが共有される。Rocky Linux 8.10環境では、NVIDIA
    の公式ドライバやocl-icd等のライブラリを適切に設定することで、これらの機能を利用可能にな
    る 11。

インターアクト（OpenGLオーバーレイ）
プラグインがホストのビューワー上に独自の操作ハンドルや描画を行いたい場合、「インターアクト（
Interacts）」機能を使用する 1。これはOpenGLを利用してフレーム上にグラフィックスをオーバーレイ
する仕組みである。

kOfxInteractActionDrawアクションが発生すると、ホストは適切なOpenGLコンテキストをカレントに
し、プラグインに描画を許可する 24。また、マウスの移動（kOfxInteractActionPenMotion）やクリック
（kOfxInteractActionPenDown）などのイベントを処理することで、画面上のハンドルをドラッグして
パラメータを操作するような直感的なUIを提供できる 1。


Rocky Linuxにおけるパッケージングとデプロイメント
開発したプラグインを配布・利用するためには、OpenFXの標準的なディレクトリ構造に従ってパッ
ケージングする必要がある。

###.ofx.bundleの構造

LinuxにおけるOpenFXプラグインは、単一の共有ライブラリファイルではなく、特定のディレクトリ構
造を持つ「バンドル（Bundle）」として提供される 35。

/usr/OFX/Plugins/MyEffect.ofx.bundle/

└── Contents/

├── Linux-x86_64/

│ └── MyEffect.ofx (実際の共有ライブラリ)

└── Resources/

├── MyEffect.png (アイコン)

└── MyEffect.xml (追加のリソース)

この構造により、一つのバンドル内に複数のアーキテクチャ（x86_64, aarch64等）のバイナリを含め
ることが可能となる 35。バイナリの拡張子は、Linux上であっても.ofxとするのが規格上の規定である

シンボル可視性とリンクの最適化
LinuxでのDSO開発において、不必要なシンボルの露出はロード時間の増大や名称衝突を招く。
OpenFXプラグインでは、エントリポイントとなる2つの関数以外はすべて「非表示（Hidden）」にするこ
とが推奨される 37。

GCCのコンパイルオプションとして-fvisibility=hiddenを指定し、公開すべき関数にのみ
__attribute__((visibility("default")))を付与する。これにより、ELFシンボルテーブルが整理され、
Boost.Pythonなどの巨大なライブラリをリンクしている場合でも、ロード時間を大幅に短縮できる 37。

VFX Reference Platformへの準拠
プロフェッショナルなVFX環境において、プラグインが他のライブラリ（Python, Qt, Boost等）と競合し
ないことは極めて重要である。VFX Reference Platformは、特定の年度におけるこれらの標準バー
ジョンを規定している 4。



 年度ターゲット          C++標準            glibc           Python


 CY2024           C++17            2.28            3.11 7

Rocky Linux 8.10の基本ライブラリであるglibc 2.28は、CY2024の要件を満たしており、GCC Toolset
13を利用することでC++17/20のコンパイル要件もクリアできる 3。


デバッグと診断手法
複雑なホストアプリケーション内で動作する共有ライブラリのデバッグは、スタンドアロンアプリケー
ションとは異なるアプローチが必要となる。

GDBによるホストプロセスへのアタッチ
Rocky Linux 8.10で最も一般的なデバッグ手法は、GDB（GNU Debugger）を使用して、プラグインを
ロードしているホストプロセス（Nuke, Natron, DaVinci Resolve等）にアタッチすることである 6。
1.​ ホストアプリケーションを通常通り起動する。
2.​ ps -aux | grep appnameでプロセスIDを確認する。
3.​ gdb -p PIDでアタッチし、ソースコードの適切な場所にブレークポイントを設定する。

あるいは、IDE（VSCodeやCLion）のデバッグ設定で「外部実行ファイル」としてホストを指定し、プラ
グインプロジェクトから直接起動させる方法も効率的である 40。

ofxTestLog.txtと診断出力
OpenFX C++サポートライブラリには、デバッグビルド時に詳細なログを出力する機能が組み込まれ
ている。これは通常、カレントディレクトリにofxTestLog.txtとして生成される 41。

このログには、ホストが提供するプロパティの妥当性確認（Property Validation）の結果が記録され
る。もしホストが規格外のプロパティ値を返していたり、必須のプロパティが欠落していたりする場
合、このログを確認することで原因を迅速に特定できる 43。また、標準出力へのprintfやstd::cout
は、ホストをターミナルから起動していれば直接確認することが可能である 41。


結論とエンジニアリング上の提言
OpenFXは、Rocky Linux 8.10という安定したエンタープライズ環境において、その真価を発揮する強
力な標準規格である。規格自体はC言語ベースのシンプルなインターフェースを採用しているが、そ
の背後にあるマルチスレッド管理、座標系変換、GPU加速といった要素は高度な専門知識を要求す
る。

開発にあたっては、まずGCC Toolsetによる現代的な開発環境を構築し、VFX Reference Platform
のガイドラインを遵守することが、安定した商用プラグインへの第一歩となる。また、単なる画像変換
処理にとどまらず、インターアクト機能を通じた直感的な操作性の提供や、OpenMfxのような3Dメッ
シュ処理への将来的な拡張を視野に入れることで、より付加価値の高いツール開発が可能となる 1。

本ガイドが、Rocky Linux 8.10におけるOpenFXプラグイン開発の強固な基盤となり、ビジュアル・エ
フェクト業界のさらなる技術革新に寄与することを期待する。
